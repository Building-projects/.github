# Building Projects

## 🧠 Applied Projects Parallel Roadmap

Esta organización reúne la ruta **Building Projects 2.0**.

Esta ruta acompaña directamente a:

```txt id="bp-path"
Path-AI-Engineer-2.0
```

Building Projects no reemplaza AI Engineer.

Building Projects acompaña AI Engineer con proyectos más pequeños, visibles, terminables y orientados a evidencia aplicada.

---

# 🎯 Objetivo general

Construir una ruta paralela de proyectos propios que convierta el avance técnico de AI Engineer en evidencia visible.

Mientras AI Engineer construye:

```txt id="bp-ai-focus"
profundidad técnica
→ sistemas completos
→ APIs
→ modelos
→ tests
→ cloud
→ MLOps
→ agentes
→ Quantum AI
→ robótica
→ plataformas finales
```

Building Projects construye:

```txt id="bp-building-focus"
microproductos
→ dashboards
→ visualizadores
→ reportes
→ demos
→ portales
→ herramientas pequeñas
→ documentación
→ capturas
→ evidencia visible
```

---

# 🧭 Filosofía de trabajo

Regla central:

```txt id="bp-philosophy"
AI Engineer = profundidad técnica.
Building Projects = aplicación visible.

AI Engineer = sistemas grandes.
Building Projects = herramientas terminables.

AI Engineer = dominio de ingeniería.
Building Projects = evidencia pública del dominio.
```

Building Projects no debe competir con AI Engineer.

Debe traducir partes importantes de AI Engineer en herramientas pequeñas y publicables.

---

# 🔗 Regla de match con AI Engineer

Building Projects hará match con los proyectos impares de AI Engineer.

La regla será:

```txt id="bp-match-rule"
Proyecto 1 IA  → Proyecto 1 Building
Proyecto 2 IA  → Nada
Proyecto 3 IA  → Proyecto 2 Building
Proyecto 4 IA  → Nada
Proyecto 5 IA  → Proyecto 3 Building
Proyecto 6 IA  → Nada
...
Proyecto 71 IA → Proyecto 36 Building
Proyecto 72 IA → Nada
```

Esto significa:

```txt id="bp-match-meaning"
Cada Building Project acompaña un proyecto específico de AI Engineer.
No dura lo mismo que dos proyectos de IA.
No se infla para cubrir huecos.
No se convierte en proyecto grande.
```

Cada Building Project toma como referencia la duración del proyecto IA con el que hace match.

Ejemplo:

```txt id="bp-match-example"
IA Proyecto 01 dura 4 semanas
→ Building Proyecto 01 dura 4 semanas

IA Proyecto 03 dura 4 semanas
→ Building Proyecto 02 dura 4 semanas

IA Proyecto 05 dura 5 semanas
→ Building Proyecto 03 dura 5 semanas
```

Pero la ruta Building Projects avanza de forma continua, sin semanas vacías internas.

# 🧩 Sistema de organización

## Proyecto propio aplicado

Un proyecto propio aplicado es un repositorio pequeño o mediano que demuestra una habilidad concreta.

Puede ser:

* dashboard;
* mini app;
* notebook visual;
* generador de reportes;
* portal ligero;
* visualizador;
* calculadora;
* buscador;
* asistente pequeño;
* tracker;
* plantilla técnica;
* herramienta de documentación;
* demo de arquitectura.

Debe cerrar con:

* README;
* capturas;
* outputs;
* demo local o evidencia;
* documentación;
* limitaciones;
* aprendizajes;
* siguiente paso;
* conexión con AI Engineer.

---

## Módulo

Un módulo es una parte funcional dentro del proyecto.

Ejemplo:

```txt id="bp-module-example"
Módulo: Insight Cards
```

Significa:

```txt id="bp-module-meaning"
Tomo resultados técnicos.
Los convierto en tarjetas visuales.
Explico qué significan.
Hago que el usuario entienda el valor.
```

---

## Lab

Un lab es un experimento pequeño y cerrado.

En Building Projects los labs deben ser:

```txt id="bp-labs"
pequeños
visibles
documentados
comparativos
terminables
```

Tipos de labs:

```txt id="bp-lab-types"
tec-labs
cloud-labs
product-labs
biz-labs
research-labs
```

# 🧪 Tipos de proyectos

## Microproducto

Herramienta pequeña con valor visible.

Ejemplos:

* dashboard de ventas;
* buscador semántico;
* calculadora de escenarios;
* generador de reportes.

---

## Visual Tool

Herramienta para explicar conceptos difíciles con tarjetas, diagramas, gráficos o tablas.

Ejemplos:

* quantum concept cards;
* neural network explainer;
* RL policy visualizer;
* humanoid architecture cards.

---

## Demo Portal

Portal ligero que agrupa información, proyectos o resultados.

Ejemplos:

* roadmap portal;
* AI lab portal;
* robotics simulation notes portal;
* portfolio platform.

---

## Report Generator

Herramienta que convierte inputs en reportes.

Ejemplos:

* model quality report;
* PQC risk checklist;
* RAG quality checker;
* generative output evaluation board.

# 📁 Estructura general recomendada

Cada proyecto puede seguir esta estructura ligera:

```txt id="bp-structure"
project-name/
│
├── README.md
├── requirements.txt
├── .gitignore
│
├── data/
│   ├── raw/
│   └── processed/
│
├── src/
│   ├── core/
│   ├── analysis/
│   ├── visualization/
│   └── utils/
│
├── reports/
│   ├── figures/
│   ├── summaries/
│   └── outputs/
│
├── dashboard/
│
├── docs/
│   ├── decisions.md
│   ├── methodology.md
│   └── architecture.md
│
├── labs/
│   ├── tec-labs/
│   ├── cloud-labs/
│   ├── product-labs/
│   └── biz-labs/
│
└── scripts/
```

No todos los proyectos deben tener todas las carpetas.

Regla:

```txt id="bp-structure-rule"
La estructura debe servir al proyecto.
No el proyecto a la estructura.
```
